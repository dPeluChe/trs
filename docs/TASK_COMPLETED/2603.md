# 2026-03-18 — Phase 1: New Features + Competitor Analysis

## Context
Competitive analysis of RTK (v0.31.0), tokf, ccp, claw-compactor, QMD, and Pi Coding Agent.
Identified gaps vs RTK and implemented Phase 1 features based on findings.
Evaluated and **discarded** MCP Server integration (adds complexity without value for a unidirectional pipe tool).

## Completed

### Emoji stripping (global, default)
- Added `strip_emojis()` to `src/router/handlers/common.rs`
- Applied in `ParseHandler::read_input()` and `Router::process_stdin()`
- Strips decorative emoji (U+1Fxxx ranges + select dingbats like U+274C, U+2705, U+2728)
- Preserves functional symbols: checkmarks, crosses, bullets, warning signs (used by test runners/build tools)
- **Why**: RTK #603 found emojis confuse non-Claude LLMs causing retry loops

### `trs json` command — JSON structure without values
- New handler: `src/router/handlers/json.rs`
- Shows keys + types + array lengths, no values
- Supports `--depth N` for max depth limit
- Supports `--json` output for programmatic schema
- Long strings show length: `String[1024]`, homogeneous arrays show type: `Array[42] of Object`
- 7 unit tests

### `gh` (GitHub CLI) parsers
- `trs gh pr list` / `trs parse gh-pr` — pull request list
- `trs gh issue list` / `trs parse gh-issue` — issue list
- `trs gh run list` / `trs parse gh-run` — workflow runs with status detection
- Supports **both** TTY format (emojis + `#`) and non-TTY format (TSV from subprocess capture)
- Auto-detect in classifier + pipe syntax

### Tee system — failure output recovery
- On non-zero exit code, saves full stdout+stderr to `~/.trs/tee/<timestamp>_<cmd>.log`
- Prints hint: `[full output: path]` so agent can recover full output
- Auto-cleanup: keeps last 20 files

### Task documentation restructured
- `tars_task.md` → `docs/TASK_TODO.md` (standardized location)
- Created `docs/TASK_COMPLETED/` with README and monthly format (YYMM.md)

### Test fixes
- Fixed `test_stdin_unicode_edge_cases` — updated to expect emoji stripping behavior
- Fixed 3 pre-existing help test failures
- **Final**: 2,409 tests passing, 0 failures

### Deep competitor analysis (repos cloned via ghq)
- RTK: 64 modules, TOML filters (60+), ParseResult tiers, configurable limits, SQLite tracking, discover command
- tokf: TOML filter pipeline (8 stages), Lua scripting, eject system, verify framework, color passthrough
- claw-compactor: Ionizer (JSON sampling), LogCrunch (log folding), DiffCrunch, Neurosyntax (AST regex fallback), SemanticDedup (shingles)
- ccp: Conservative philosophy, YAML filters, command mutation (`--porcelain`), passthrough for structured output

## Decisions

### Discarded: MCP Server
trs is a unidirectional pipe/filter. MCP adds protocol overhead without reducing more tokens.

### Discarded: Cloning QMD/Pi repos (later cloned RTK/tokf/ccp/claw-compactor instead)
TypeScript repos not directly reusable. Only patterns extracted.

### Discarded (for now): TOML filter plugin system
tokf proves it works but adds significant complexity. Consider for Phase 3 when user demand exists.

### Discarded (for now): Lua scripting escape hatch
Overengineering for current stage. Revisit if TOML filters are adopted.

# 2026-03-18 (session 2) — Safety, Config, Read, Core Improvements

## Context
Continued from session 1. Implemented safety features (passthrough, fallback, git opts),
config system, `trs read` with filter levels, and compression pattern improvements
learned from competitor analysis (RTK, claw-compactor).

## Completed

### Safety & Robustness
- Passthrough inteligente: detect `--json`/`--porcelain`/`--format=json` → skip parser entirely
- 3-tier fallback: parser OK → truncated passthrough (configurable max) with `[trs:passthrough]` marker
- Git global options: strip `-C`, `--no-pager`, `--git-dir`, etc. before classifying subcommand
- `has_structured_output_flag()` and `strip_git_global_opts()` in classifier.rs

### Config system (`~/.trs/config.toml`)
- New module: `src/config.rs` with `Config` + `Limits` structs (serde + toml)
- `[limits]`: grep_max_results, grep_max_per_file, status_max_files, passthrough_max_chars, json_max_depth, tee_max_bytes, tee_max_files
- Project-local override via `.trs/config.toml`
- Wired into passthrough fallback and tee system

### `trs read` — file reader with filter levels
- New handler: `src/router/handlers/read.rs`
- 3 levels: none (raw), minimal (strip comments), aggressive (signatures-only)
- Language detection by extension: Rust, Python, JS/TS, Go, Java/C/C++, Ruby, Shell
- Data file protection: JSON/YAML/TOML/XML always passthrough unmodified
- Supports `--lines N`, `--tail N`, `--line-numbers`
- 93.7% reduction in aggressive mode on Rust files

### Log parser improvements (LogCrunch patterns)
- Fold runs of INFO/DEBUG >= 3 into first + `[...N similar lines...]` + last
- Stack trace detection: "at ", "File ", "Caused by", indented lines as atomic blocks
- ERROR/WARN/FATAL always shown individually, never folded

### Git diff improvements (DiffCrunch patterns)
- Hunk-level parsing with line tracking
- Context compression: blocks > 4 lines → first 1 + `[...N unchanged...]` + last 1
- Large diff summary: > 200 lines → prepend file list with +/-

### trs json improvements (Ionizer patterns)
- Array sampling: arrays > 5 items → first 3 + last 3 + error items preserved
- ID field detection: "id", "uuid", "_id", "*_id" annotated with `(id)`
- Non-JSON rejection with hints (suggests `trs read` for .toml/.yaml)
- 19 new tests

## Decisions
- Config uses TOML (not YAML) — matches Rust ecosystem (Cargo.toml)
- Limits defined but not yet wired into all parsers (grep, git status) — future task
- `trs read` uses regex fallback (no tree-sitter) — simpler, zero deps, works for 8 languages

# 2026-03-19 (session 4) — Competitive improvements + generic compression

## Completed

### git status directory grouping (>20 files)
- When a section has >20 entries, groups by parent directory with status counts
- Example: `src/router/ (M:10 A:2 D:1)` instead of listing 12 files individually
- Result: 159B vs rtk 245B on 35-file scenario (35% smaller)

### gh pr/issue/run title truncation
- PR titles: max 60 chars, authors: max 30 chars, run names: max 50 chars
- Full values preserved in `--json` output
- Closed gap vs rtk: 471B vs 466B (was 565B vs 466B)

### Compact git push/pull/fetch parser
- Strips remote URL, shows only ref range and branch
- `pushed ae7dfe3..d6fd77d main -> main` instead of full URL
- Handles up-to-date, fast-forward, new branch, errors
- 7 unit tests

- Commands without a dedicated parser now get lightweight compression
- Collapses whitespace padding, strips ANSI codes, removes progress bars
- Result: `ollama list` 1032B → 626B (39% reduction) with zero config
- Benefits any tabular output: kubectl, aws, ollama, etc.
